# 竞品 AI 合约提取框架

> 本文档在 Layer 4（AI 合约提取）时加载。仅当竞品使用 AI 功能时需要。
> 来源: CC/CD Framework, Capability-First Design, AI Product Contract Patterns

## AI 功能识别

### 如何发现竞品的 AI 功能

| 信号 | 检查方式 |
|------|---------|
| 官网提及 "AI" / "智能" / "自动生成" | WebFetch 首页 + 功能页 |
| 定价页标注 "AI credits" / "AI 额度" | WebFetch 定价页 |
| 产品演示中有生成/推荐功能 | WebSearch "{竞品} AI features / demo" |
| 用户评价提及 AI 质量 | WebSearch "{竞品} AI quality review" |
| 技术博客/API 文档提及模型 | WebSearch "{竞品} engineering blog AI model" |

### AI 功能分类

| 类别 | 说明 | 典型功能 |
|------|------|---------|
| **生成类** | AI 从无到有创建内容 | 文案生成、图片生成、代码生成 |
| **转换类** | AI 将 A 格式转为 B 格式 | 摘要、翻译、格式转换 |
| **推荐类** | AI 基于数据给出建议 | 产品推荐、定价建议、内容策略 |
| **分析类** | AI 从数据中提取洞察 | 用户行为分析、趋势识别 |
| **对话类** | AI 与用户进行交互式对话 | 客服、共创、引导式配置 |

## 可靠度评估

### 三级分级（从外部观察）

| 级别 | 可靠度 | 外部观察信号 |
|------|--------|-------------|
| **可靠** | >95% | AI 输出直接使用无需编辑；用户评价普遍正面；功能默认开启 |
| **中等** | 70-95% | AI 输出需要用户编辑/选择；有"重新生成"按钮；评价褒贬不一 |
| **实验** | <70% | 标注为"Beta"；输出仅作建议；有完整的手动 fallback |

### 评估时的注意事项

- **不要高估**: 官网宣传的 AI 能力通常比实际体验好。以用户评价和实际演示为准。
- **关注边界**: AI 做不到什么比做得到什么更有价值。
- **看降级**: 没有 AI 降级方案的产品 = AI 功能是核心依赖 = 风险集中点。

## AI 合约提取方法

### 为什么要提取"合约"而非只列"功能"

功能列表只回答"竞品有什么 AI"，合约回答"AI 是怎么工作的"。当你设计自己的 AI 产品时，你需要知道：

- 竞品的 AI 需要什么输入？（你可能需要类似的数据）
- 竞品的 AI 产出什么格式？（用户已有这个预期）
- 竞品觉得什么质量"够好"？（行业基准）
- 竞品 AI 失败时怎么办？（降级方案的行业共识）

### 合约四维度提取

对每个 AI 功能，按以下四个维度提取：

#### 维度 1: 输入（Input）

| 提取项 | 含义 | 信息来源 |
|--------|------|---------|
| 必需输入 | AI 最少需要什么才能工作 | UI 表单必填字段、Onboarding 流程 |
| 可选输入 | 有了更好的额外信息 | UI 可选字段、高级设置 |
| 上下文来源 | 从哪获取额外信息 | 社媒导入、历史数据、第三方 API |
| 用户提供 vs 系统获取 | 哪些信息需要用户输入 | 表单字段 vs 自动填充 |

外部观察方法：
```
1. 注册/Onboarding 流程中问了什么 → 必需输入
2. 可跳过的步骤问了什么 → 可选输入
3. 有"连接社媒"/"导入数据"功能 → 上下文来源
4. 有表单 vs 只需点击 → 用户负担程度
```

#### 维度 2: 输出（Output）

| 提取项 | 含义 | 信息来源 |
|--------|------|---------|
| 输出格式 | 产出什么——文本/图片/结构化数据 | 产品演示、功能页截图 |
| 输出数量 | 一次生成几个选项 | UI 展示方式（单结果 vs 多选项卡片） |
| 可编辑性 | 用户能否修改 AI 输出 | 有无编辑器/表单 |
| 输出结构化程度 | 是否有明确字段 | 结构化卡片 vs 纯文本 |

外部观察方法：
```
1. AI 生成后展示什么 → 输出格式
2. 展示 1 个还是 2-3 个方案 → 输出数量
3. 有"编辑"按钮/富文本编辑器 → 可编辑
4. 生成结果是卡片/表格/文本 → 结构化程度
```

#### 维度 3: 质量线（Quality）

| 提取项 | 含义 | 信息来源 |
|--------|------|---------|
| 可靠度等级 | >95%/70-95%/<70% | 见上方"三级分级" |
| 质量信号 | 用户评价中的质量反馈 | 用户评价、社媒讨论 |
| 重新生成率暗示 | "重新生成"按钮是否突出 | UI 设计（按钮位置/大小） |
| 编辑率暗示 | 用户通常需要编辑多少 | 用户评价、教程视频 |

外部观察方法：
```
1. "重新生成"按钮多大多明显 → 暗示输出不稳定
2. 编辑器功能多丰富 → 暗示用户常需修改
3. 用户评价中"amazing"vs"需要调整" → 质量分布
4. 有无"AI 生成"标签 → 产品对 AI 质量的自信度
```

#### 维度 4: 降级方案（Fallback）

| 提取项 | 含义 | 信息来源 |
|--------|------|---------|
| AI 失败体验 | 用户看到什么——空白/错误/默认内容 | 产品边缘情况测试 |
| 手动替代路径 | 能否完全绕过 AI | 有无"手动创建"/"跳过 AI"选项 |
| 核心 vs 增强 | 去掉 AI 产品能不能用 | 产品架构分析 |
| 超时/限流处理 | 高并发时怎么办 | 用户评价中的延迟抱怨 |

外部观察方法：
```
1. 有无"手动创建"/"从模板开始"选项 → 手动替代路径
2. AI 生成中 loading 时间 + 有无取消按钮 → 超时处理
3. 定价页有无 "AI 额度限制" → 限流机制
4. 去掉 AI 后产品还有什么功能 → 核心依赖程度
```

### 合约提取模板

对每个 AI 功能，填充此模板：

```yaml
AI 功能: {功能名称}
类别: 生成/转换/推荐/分析/对话

输入:
  必需: {最少需要什么}
  可选: {有了更好的额外信息}
  来源: {用户手动输入 / 社媒导入 / 系统数据 / 第三方API}
  用户负担: 低(只需点击) / 中(填几个字段) / 高(需要详细描述)

输出:
  格式: {文本 / 图片 / 结构化数据 / 选项列表 / 页面}
  数量: {1个 / 2-3个选项 / 批量N个}
  可编辑: 是(富文本) / 是(表单) / 部分 / 否
  交付: 即时 / 异步(N秒/分钟)

质量线:
  可靠度: 可靠(>95%) / 中等(70-95%) / 实验(<70%)
  衡量信号: {用户评价 / 重新生成按钮显著性 / 编辑器复杂度}
  证据: {具体观察到的信号}

降级方案:
  AI失败时: {空白页 / 默认模板 / 手动模式 / 重试提示}
  绕过路径: {有手动替代 / 无法绕过}
  核心依赖: 核心(没有AI产品不可用) / 增强(没有AI也能用,体验降级) / 可选(锦上添花)

来源: [产品演示 / 用户评价 / API文档 / 定价页 / 推测]
```

## HITL (Human-in-the-Loop) 设计观察

### 竞品的 HITL 模式

观察竞品在 AI 输出后如何让用户参与：

| HITL 等级 | 用户参与度 | 产品表现 |
|-----------|-----------|---------|
| **自动** | AI 输出直接生效 | 用户可事后查看/修改 |
| **审核** | AI 输出后需确认 | "确认" / "编辑"按钮 |
| **选择** | AI 提供多选项 | 2-3 个方案卡片 |
| **协作** | AI 草稿 + 用户编辑 | 富文本编辑器 + AI 建议 |
| **灵感** | AI 仅提供参考 | 侧边栏建议，用户从头创建 |

### 确认点密度分析

确认点 = AI 输出后需要用户确认的位置。关键分析维度：

```
确认点密度 = 确认点数量 / AI 功能总数

密度过低（< 0.3）: 用户可能感觉失控
密度过高（> 0.8）: AI 省事的价值被稀释
合理区间（0.4-0.7）: 关键节点确认，次要环节自动
```

观察方法：
- 画出竞品的 AI 交互流程图
- 标注每个确认点的位置
- 计算整体密度
- 分析哪些确认点是"必要的"（涉及金钱/发布），哪些是"防御性的"（怕 AI 出错）

### AI 确认点链模式

```
[AI 生成方向] → [用户确认] → [AI 生成产品] → [用户确认] → [AI 生成页面] → [用户确认] → [上线]
```

关键观察：
- 确认点之间有没有回退路径（选了方向能不能回去重选）
- 确认的粒度（整体确认 vs 逐字段确认）
- 拒绝后的处理（重新生成 vs 手动编辑 vs 放弃）
