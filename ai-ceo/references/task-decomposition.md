# 任务拆解方法论

> CEO 目标拆解的核心参考文档。用于 PLAN 模式 Phase 2 的任务分解。

---

## 1. 四维度框架

每个公司目标必须从 4 个维度进行拆解，确保执行的完整性：

### 产品维度 (P)

**范围**：直接面向用户/客户的功能和体验

- 核心功能开发与迭代
- 用户体验优化
- 产品定价与包装
- 产品文档与帮助系统
- 数据分析仪表盘（面向客户）

**典型任务举例**：
- 完成 MVP 核心功能开发
- 上线用户反馈收集机制
- 设计并验证定价方案
- 完成产品 demo 视频

**负责人**：CPO

### 增长维度 (G)

**范围**：用户获取、激活、留存、变现、推荐（AARRR）

- 获客渠道建设与测试
- 内容营销与 SEO
- 社交媒体运营
- 付费广告投放
- 合作伙伴与渠道
- 用户激活与留存优化
- 品牌建设

**典型任务举例**：
- 完成 3 个获客渠道 MVP 测试
- 搭建内容发布 pipeline
- 完成首批 KOL 合作
- 设计新用户 onboarding 流程

**负责人**：CMO

### 技术维度 (T)

**范围**：支撑产品和增长的技术基础设施

- 架构设计与搭建
- 核心服务开发
- DevOps 与部署流水线
- 数据基建（埋点、分析、存储）
- 安全与合规
- 性能优化
- 第三方服务集成

**典型任务举例**：
- 搭建 CI/CD 流水线
- 完成核心 API 开发
- 接入支付系统
- 完成数据埋点方案并实施

**负责人**：CTO

### 运营维度 (O)

**范围**：保障业务运转的日常运营和管理

- 客户服务与支持
- 财务与法务
- 团队招聘与管理
- 流程制度建设
- 供应商与外包管理
- 数据报表与汇报

**典型任务举例**：
- 建立客户支持 SOP
- 完成公司注册/资质办理
- 搭建团队协作工具链
- 制定月度汇报流程

**负责人**：COO

## 2. MECE 拆解原则

**MECE** = Mutually Exclusive, Collectively Exhaustive（相互独立，完全穷尽）

### 核心规则

1. **相互独立**：任意两个任务之间没有重叠。如果两个任务有部分重叠，要么合并，要么重新划分边界。
2. **完全穷尽**：所有任务合在一起，完整覆盖了目标的实现路径。检查方法：如果所有任务都完成了，目标是否一定达成？如果不是，说明遗漏了任务。

### 拆解方法

**自上而下拆解（推荐）**：
1. 从目标出发，问："要实现这个目标，必须完成哪几件事？"
2. 对每件事，再问："要完成这件事，必须完成哪几个子任务？"
3. 重复直到粒度达标（1-2 周可完成）

**自下而上验证**：
1. 列出所有能想到的任务
2. 按维度归类
3. 检查：合在一起是否覆盖目标？有没有重叠？

### 常见错误

| 错误 | 示例 | 修正 |
|------|------|------|
| 任务重叠 | "优化首页" 和 "提升转化率" 有交叉 | 按页面拆分，或按漏斗阶段拆分 |
| 遗漏维度 | 只拆了产品和技术，没有增长和运营 | 用 4 维度框架检查 |
| 粒度不一 | "搭建整个后端" vs "修改一个按钮颜色" | 统一到 1-2 周粒度 |
| 缺少验证任务 | 只有"开发"没有"验证" | 为每个关键假设添加验证任务 |

## 3. 任务粒度标准

### 合格任务的 5 个条件

1. **时间范围**：1-2 周内可完成。超过 2 周的必须继续拆分。
2. **可衡量结果**：有明确的完成标准（数字、交付物、状态变化）。"做了" 不算，"做到什么程度" 才算。
3. **单一负责人**：一个任务只能有一个 owner。需要多人协作的，拆成多个任务并标注依赖。
4. **明确输入输出**：开始时需要什么（依赖），完成后产出什么（交付物）。
5. **可独立验收**：不依赖其他任务也能判断是否完成。

### 粒度参考

| 太粗（需拆分） | 合适 | 太细（需合并） |
|---------------|------|--------------|
| 完成 MVP 产品开发 | 完成用户注册+登录功能 | 写注册页的 CSS |
| 搭建营销体系 | 完成 Twitter 账号冷启动（50 个种子关注） | 发一条推文 |
| 技术架构设计 | 完成 API 接口文档 + 核心 3 个端点开发 | 修改一个环境变量 |

## 4. 依赖关系处理

### 依赖类型

| 类型 | 定义 | 处理方式 |
|------|------|---------|
| **硬依赖（Blocking）** | A 不完成，B 无法开始 | 必须在计划中体现先后顺序 |
| **软依赖（Non-blocking）** | A 完成后 B 更高效，但 B 可以先开始 | 标注但不阻塞排期 |
| **资源依赖** | A 和 B 需要同一个人/资源 | 错开排期或找替代资源 |

### 关键路径识别

关键路径 = 从开始到目标达成的最长依赖链路。

识别方法：
1. 画出所有任务的依赖关系图
2. 找出从起点到终点的所有路径
3. 最长的那条就是关键路径
4. 关键路径上的任何延迟都会导致整体延迟

**关键路径上的任务必须是 P0**，资源优先保障。

### 依赖标注格式

```
dependencies: [G1-P-01, G1-T-02]   # 硬依赖
soft_deps: [G1-G-01]               # 软依赖
```

## 5. 优先级排序标准

### P0 / P1 / P2 定义

| 级别 | 定义 | 数量限制 |
|------|------|---------|
| **P0** | 不做就无法达成目标的核心任务 | 每维度 ≤ 3 个 |
| **P1** | 做了能显著加速目标达成 | 每维度 ≤ 5 个 |
| **P2** | 锦上添花，资源充足时做 | 不限制 |

### 排序维度（按权重）

1. **主要矛盾相关性（40%）**：是否直接解决 CEO 诊断中识别的主要矛盾？
2. **收入/增长影响（30%）**：对收入或用户增长的直接影响有多大？
3. **风险（20%）**：不做的风险有多大？会不会导致不可逆的损失？
4. **投入产出比（10%）**：需要多少资源？回报是否值得？

### 快速排序法

对每个任务问 3 个问题：
1. "如果这个月只能做 3 件事，这个会是其中之一吗？" → 是 = P0 候选
2. "如果不做这个，目标会失败吗？" → 是 = P0
3. "这个能等到下个月吗？" → 能 = P1 或 P2

## 6. 公司阶段判断标准

不同阶段的公司，任务优先级和策略有本质差异。

### 冷启动阶段 (0→1)

**判断条件**（满足任意 2 个）：
- 月活用户 < 100
- 月收入 < 1 万
- 产品核心功能未验证
- 没有付费用户

**战略重心**：
- 产品：聚焦核心功能，快速迭代，收集反馈
- 增长：种子用户获取，手动不规模化
- 技术：最小基建，速度优先，技术债可接受
- 运营：创始人亲自做，不建体系

**P0 任务特征**：
- 直接验证 PMF 的任务
- 能带来前 100 个种子用户的任务
- 实现第一笔收入的任务

### 增长阶段 (1→10)

**判断条件**（满足任意 2 个）：
- 月活用户 100-10,000
- 月收入 1 万-100 万
- 有明确的付费用户群
- 产品核心价值已验证

**战略重心**：
- 产品：功能完善，提升体验，建立壁垒
- 增长：找到可规模化的渠道，建立增长飞轮
- 技术：补还技术债，建设数据基建，提升稳定性
- 运营：建立核心 SOP，开始团队分工

**P0 任务特征**：
- 放大已验证渠道的任务
- 提升留存和 NPS 的任务
- 建立自动化流程的任务

### 成熟阶段 (10→100)

**判断条件**（满足任意 2 个）：
- 月活用户 > 10,000
- 月收入 > 100 万
- 市场份额稳定
- 团队 > 20 人

**战略重心**：
- 产品：平台化，多产品线，生态建设
- 增长：品牌建设，新市场拓展，精细化运营
- 技术：架构升级，安全合规，AI/自动化
- 运营：流程标准化，人才梯队，组织文化

**P0 任务特征**：
- 防御性壁垒建设的任务
- 开拓第二增长曲线的任务
- 组织能力建设的任务
