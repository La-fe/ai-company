> 来源: Shape Up (Basecamp), CC/CD Continuous Calibration, BVP AI PMF

## Cupcake 定义法

Cupcake = 最小端到端闭环。不是最简功能列表，而是**用户首次感知到价值的最短路径**。

### 识别步骤

1. **画出端到端旅程**: 用户从"进入产品"到"获得价值"的完整路径
2. **标注每步的最简实现**: 能用 ≠ 好用，只要能走通
3. **删到只剩"价值首次可感知"**: 删除所有"锦上添花"步骤
4. **检查闭环**: 剩余步骤能否形成完整闭环（有输入→有处理→有输出→有价值）
5. **Feature 数量检查**: Cupcake ≤ 6 Feature

### Cupcake 检查清单

| 检查项 | 合格标准 |
|--------|---------|
| Feature 数量 | ≤ 6 个 |
| 端到端可用 | 用户能从头走到尾，不会卡在中间 |
| 价值可感知 | 用户走完后会说"这有用"而非"然后呢？" |
| 无外部依赖 | 不依赖用户先做 N 步准备工作 |
| 首次使用时间 | < 10 分钟从注册到价值感知 |

## Appetite 约束

Shape Up 的核心理念：**先定投入（appetite），再定范围**。

### Appetite 思维

传统做法：设计功能 → 估算时间 → 发现太久 → 砍功能（痛苦）

Appetite 做法：**"这个问题值得我们花多少时间？"** → 在这个时间内能做的 = 范围

### Appetite 设定参考

| Appetite | 适用场景 | 期望产出 |
|---------|---------|---------|
| **1 周** | 小改进、bug 修复、UI 调整 | 单个功能点的完善 |
| **2 周** | 新功能模块、中等复杂度集成 | 一个完整的功能闭环 |
| **6 周** | 新产品线、大型功能集 | 一个完整的 Cupcake |
| **不做** | 投入/产出比不合理 | — |

### 范围适配原则

```
Appetite = 2 人 × 4 周
功能列表超了？→ 不加时间，削范围
范围削不动？→ 说明 Appetite 设错了（问题定义需要重新审视）
```

## 兔子洞识别

兔子洞（Rabbit Hole）= 看起来简单但实际上可能吞噬大量时间的功能点。

### AI 产品常见兔子洞

| 兔子洞 | 为什么危险 | 识别信号 | 应对策略 |
|--------|-----------|---------|---------|
| **第三方集成** | 每个平台 API 不同，文档不全，频繁变更 | "接入 Stripe/Shopify/社媒 API" | 先接最关键的 1 个，其余延后 |
| **AI 输出质量调优** | Prompt Engineering 是无底洞 | "让 AI 输出更好" 没有明确的"够好"标准 | 先定义"可接受"标准，达到就停 |
| **多平台适配** | 每个平台规则不同（字数、格式、审核） | "同时支持微信/抖音/小红书" | 先做 1 个平台，验证后复制 |
| **实时协作** | WebSocket、冲突解决、状态同步 | "多人同时编辑" | MVP 用锁机制（同时只能 1 人编辑） |
| **完美 Onboarding** | 引导流程设计是另一个产品 | "智能引导新用户" | MVP 用静态教程，V1.1 再优化 |
| **自定义工作流** | 工作流引擎 = 一个独立产品 | "用户自定义自动化流程" | MVP 提供预设模板，不开放自定义 |

### 兔子洞评估模板

```markdown
| Feature | 表面复杂度 | 实际复杂度 | 兔子洞风险 | 应对策略 |
|---------|-----------|-----------|-----------|---------|
| {功能A} | 简单 | 中等 | 低 | 正常开发 |
| {功能B} | 简单 | 极复杂 | 高 | 限制范围: {具体限制} |
| {功能C} | 中等 | 中等 | 低 | 正常开发 |
```

## 风险评估快速清单（6 类 AI 特有风险）

| 类型 | 典型场景 | 检测方式 | 缓解策略 |
|------|---------|---------|---------|
| **质量风险** | AI 推荐方向不靠谱 | 用户弃选率、负面反馈率 | 降级为推荐热门品类；增加人工审核 |
| **信任风险** | 用户按 AI 建议做但失败 | 退款率、差评率、客服工单 | 明确免责声明 + 概率提示（"建议仅供参考"） |
| **经济风险** | API 成本超过收入 | 成本监控仪表盘、毛利率追踪 | 限流 + 缓存 + 模型降级 + 调整定价 |
| **法律风险** | 生成内容侵权/违规 | 相似度检测、合规审查工具 | 原创性 filter + 内容审核 + 用户协议免责 |
| **偏见风险** | 只推荐某类产品/方向 | 输出分布分析、A/B 对比 | 多样性约束 + 定期审计输出分布 |
| **依赖风险** | API 涨价/下线/降质 | 供应商监控、多 Provider 测试 | 多供应商 + 降级方案 + 模型抽象层 |

## 持续校准节奏

AI 产品不是"做完上线就结束"，而是需要持续校准质量。

### 校准节奏

| 阶段 | 节奏 | 聚焦 | 原因 |
|------|------|------|------|
| **首月** | 每周跑评估 | 发现核心质量问题 | 产品刚上线，数据少但变化快 |
| **2-3 个月** | 双周跑评估 | 追踪质量趋势 | 数据积累，需要发现模式 |
| **稳定期** | 每月跑评估 | 防止质量退化 | 模型更新、用户变化可能导致退化 |
| **模型升级后** | 立即跑评估 | 回归测试 | 新模型可能行为不同 |

### 涨标追踪

**核心洞察：用户对 AI 质量的期望每月都在涨。同样的质量，3 个月后 = 体验下降。**

```
Month 1: AI 生成的描述"还行" → 用户满意度 4.2/5
Month 3: 同样质量的描述 → 用户满意度 3.8/5（因为用户见过更好的）
Month 6: 同样质量 → 用户满意度 3.5/5（竞品也在进步）
```

追踪方法：
- 固定评估数据集，每月跑一次，记录质量分数变化
- 用户满意度 NPS 每月追踪
- 竞品 AI 质量定期对比

### 反馈→规格回流机制

用户的每次"不满意"都是改进 AI 的信号：

```
用户拒绝 AI 输出
    ↓
收集: 输入 + AI 输出 + 用户操作（编辑/重新生成/手动完成）
    ↓
分析模式: 哪类输入经常被拒绝？拒绝原因是什么？
    ↓
更新: Prompt / 质量标准 / 评估数据集
    ↓
验证: 新版本在评估集上跑回归
    ↓
发布: 灰度 → 全量
```
