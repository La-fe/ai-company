# PRD 方法论参考手册

> 本文档为 AI CPO 的参考资料，在 Phase 2（PRD 生成）时按需加载。

---

## PRD 写作标准

一份好的 PRD 需要满足以下标准：

### 核心原则

1. **问题优先**：先清晰定义问题，再讨论解决方案。如果问题定义不清晰，功能再多也是浪费。
2. **用户具体化**：不写"用户"，写"月 GMV 10-50 万的跨境电商独立站运营"。越具体，功能设计越精准。
3. **可衡量的成功**：每个功能都要回答"上线后怎么知道它成功了"。不能衡量的功能不写进 PRD。
4. **取舍透明**：明确写出"我们不做什么"以及"为什么不做"。
5. **面向行动**：PRD 读完后，工程师和设计师能直接开始工作，而不是还需要开会澄清。

### 质量检查清单

| 维度 | 合格标准 | 常见问题 |
|------|---------|---------|
| 问题定义 | 能用一句话说清楚要解决的核心痛点 | 问题太宽泛、混入了解决方案 |
| 用户画像 | 有具体的角色、场景、痛点描述 | 只写了"用户"、没有场景 |
| 用户故事 | 每个故事都有角色、行动、收益、验收标准 | 故事太抽象、缺少验收标准 |
| 功能列表 | 每个功能有明确的 P0/P1/P2 分级和理由 | 全是 P0、没有取舍 |
| 非功能需求 | 有具体的性能、安全、兼容性要求 | 只写了"要快"、没有量化 |
| 成功指标 | 每个指标有基线值、目标值、测量方式 | 指标不可测量、没有基线 |
| 竞品分析 | 有客观的比较矩阵，不只是"我们更好" | 只罗列竞品、没有洞察 |

## 用户故事写法

### 标准格式

```
作为 [具体角色]，
我想要 [具体行动]，
以便 [可衡量的收益]。
```

### 写作规则

1. **角色要具体**
   - 不好："作为一个用户"
   - 好："作为一个每天需要上架 20+ SKU 的跨境电商运营"

2. **行动要可操作**
   - 不好："我想要更好地管理产品"
   - 好："我想要批量导入 CSV 并自动生成多语言产品描述"

3. **收益要可衡量**
   - 不好："以便提高效率"
   - 好："以便将单个 SKU 的上架时间从 30 分钟缩短到 5 分钟"

### 验收标准写法

每个用户故事必须附带验收标准（Acceptance Criteria）：

```
验收标准：
- [ ] 用户上传 CSV 后，系统在 30 秒内解析并显示预览
- [ ] 自动生成的描述覆盖英语、德语、法语三种语言
- [ ] 用户可以编辑任意生成的描述后一键发布
- [ ] 单次批量导入支持最多 500 个 SKU
```

验收标准的要求：
- 使用 checkbox 格式，方便跟踪
- 每条标准独立可验证
- 包含边界条件（最大值、最小值、异常情况）
- 避免模糊词汇（"快速"、"友好"、"良好"）

### 用户故事优先级判断

| 问题 | 回答"是" → | 回答"否" → |
|------|-----------|-----------|
| 没有它，用户能完成核心任务吗？ | 不是核心 | P0 候选 |
| 用户第一天就需要它吗？ | P0 候选 | 可以延后 |
| 竞品都有这个功能吗？ | 可能是 P0（桌面赌注） | 可能是差异化机会 |
| 它的开发成本超过 2 周吗？ | 考虑拆分或延后 | 可以进入 MVP |

## 功能分级标准

### P0 — MVP 必备

**定义**：没有它，产品不成立；用户无法完成核心任务。

**判断方法**：
- 删除这个功能后，用户还能从产品中获得核心价值吗？
- 如果不能 → P0
- 如果可以但体验打折 → P1

**典型特征**：
- 用户注册/登录（如果需要账号体系）
- 核心业务流程的最短路径
- 数据输入和核心输出
- 最基本的错误处理

**数量控制**：MVP 的 P0 功能建议不超过 5 个。如果超过，说明 MVP 定义不够精准，需要进一步裁剪。

### P1 — 增强功能

**定义**：显著提升用户体验，但没有它用户也能完成核心任务。

**判断方法**：
- 有了这个功能，用户满意度/效率/留存是否显著提升？
- 如果是 → P1
- 如果提升有限 → P2

**典型特征**：
- 批量操作（单个操作在 P0 中已有）
- 数据导出/报表
- 自定义设置
- 更好的搜索和筛选
- 快捷方式和效率工具

### P2 — 锦上添花

**定义**：提供竞争优势或差异化体验，但不影响核心流程。

**判断方法**：
- 这个功能是否会成为用户选择我们而非竞品的原因？
- 如果是 → 考虑提升到 P1
- 如果只是"有了更好" → P2

**典型特征**：
- 社交分享/邀请机制
- 高级分析和洞察
- 个性化/AI 推荐
- 第三方集成（非核心）
- 品牌化/白标功能

## 需求验证方法

在写入 PRD 之前，关键假设需要验证：

### 验证层级

| 层级 | 方法 | 成本 | 信心度 |
|------|------|------|--------|
| L1: 桌面研究 | 竞品分析、行业报告、用户论坛 | 低 | 30% |
| L2: 用户对话 | 5-10 个目标用户访谈 | 中 | 60% |
| L3: 原型测试 | 低保真原型 + 用户测试 | 中 | 75% |
| L4: 灰度验证 | 最小功能上线 + 数据监控 | 高 | 90% |

### 关键假设清单

在 PRD 中明确列出需要验证的假设：

```markdown
## 关键假设（待验证）

| # | 假设 | 验证方式 | 验证状态 |
|---|------|---------|---------|
| H1 | 目标用户愿意为 [功能] 付费 | 用户访谈 + 定价测试 | 待验证 |
| H2 | [场景] 是高频使用场景 | 竞品数据 + 用户日志 | 已初步验证 |
| H3 | [技术方案] 可以达到 [性能要求] | 技术 POC | 待验证 |
```

### 验证优先级

先验证"致命假设"——如果它错了，整个产品方向需要调整的假设。

判断标准：
1. **影响范围**：这个假设影响多少功能？影响核心流程 → 优先验证
2. **不确定性**：我们对这个假设有多少信心？信心越低 → 越需要验证
3. **验证成本**：验证这个假设需要多少时间和资源？成本低 → 先验证

## PRD 评审 Checklist

### 完整性检查

- [ ] 问题定义部分回答了"什么问题"、"为谁"、"为什么是现在"
- [ ] 用户画像具体到角色、场景、痛点（不是泛泛的"用户"）
- [ ] 每个用户故事都有验收标准
- [ ] 功能列表有明确的 P0/P1/P2 分级
- [ ] 非功能需求有量化指标
- [ ] 成功指标有基线值和目标值
- [ ] 包含竞品分析摘要
- [ ] 关键假设已列出并有验证计划

### 清晰度检查

- [ ] 新加入团队的工程师能看懂每个功能的描述
- [ ] 没有使用内部黑话或未定义的缩写
- [ ] 功能之间的依赖关系清晰
- [ ] "不做什么"部分明确列出
- [ ] 没有模糊词汇（"快速"、"良好体验"、"高性能"）

### 可行性检查

- [ ] P0 功能在预期时间内可以完成
- [ ] 技术依赖项已识别（第三方 API、基础设施等）
- [ ] 没有需要"发明新技术"才能实现的功能
- [ ] 非功能需求在现有架构下可以满足
- [ ] 关键技术风险已标注

### 可衡量性检查

- [ ] 每个成功指标都有明确的测量方式
- [ ] 数据采集方案已考虑（需要埋点吗？用什么工具？）
- [ ] 有短期指标（发布后 1 周）和中期指标（发布后 1 月）
- [ ] 指标之间不冲突（例如"提升留存"和"快速获客"可能需要平衡）
