# 版本规划方法论参考手册

> 本文档为 AI CPO 的参考资料，在 Phase 3（版本规划 + 功能拆分）时按需加载。

---

## 版本节奏设计

### 3 个月迭代周期

以 3 个月为一个完整的产品迭代周期，每个月发布一个版本：

| 版本 | 时间 | 主题 | 核心目标 |
|------|------|------|---------|
| V1.0 | Month 1 | MVP 发布 | 验证核心价值，获取第一批用户 |
| V1.1 | Month 2 | 数据驱动迭代 | 修复体验问题，验证增长假设 |
| V1.2 | Month 3 | 增长优化 | 扩大用户规模，建立竞争壁垒 |

### 每个版本的节奏

**V1.0 — MVP（第 1 个月）**

```
Week 1: 核心功能开发
Week 2: 核心功能开发 + 内部测试
Week 3: Bug 修复 + 基础文档
Week 4: 灰度发布 + 收集反馈
```

关键纪律：
- 只做 P0 功能，任何"顺手加一个"都要拒绝
- 内部测试至少覆盖所有核心流程
- 灰度先给 10-20 个种子用户

**V1.1 — 数据验证（第 2 个月）**

```
Week 1: 分析 V1.0 数据 + 规划 V1.1 范围
Week 2-3: 体验优化 + 选择性 P1 功能
Week 4: 发布 + A/B 测试关键功能
```

关键纪律：
- 先看数据，再决定做什么（不是凭感觉）
- V1.0 遗留的严重体验问题 > 新功能
- 开始建立数据采集和分析基础设施

**V1.2 — 增长优化（第 3 个月）**

```
Week 1: 增长策略对齐（与 CMO 同步）
Week 2-3: 增长功能 + 剩余 P1 + 技术债务
Week 4: 发布 + 下一个周期规划
```

关键纪律：
- 增长功能需要与 CMO 的渠道策略配合
- 预留 20% 时间处理技术债务
- 最后一周开始下一个 3 个月周期的规划

## MVP 裁剪原则

### 核心理念

> MVP 不是"功能少的产品"，而是"能验证核心假设的最小产品"。

### 裁剪决策树

对每个计划中的功能，依次问以下问题：

```
1. 删掉这个功能，用户还能完成核心任务吗？
   ├── 能 → 不进 MVP（考虑 V1.1 或 V1.2）
   └── 不能 → 继续问 ↓

2. 有没有更简单的替代方案？
   ├── 有（比如先用人工流程代替自动化）→ 用简单方案
   └── 没有 → 继续问 ↓

3. 能不能拆分成更小的功能？
   ├── 能 → 拆分，只保留最核心的部分进 MVP
   └── 不能 → 进入 P0 功能列表
```

### 常见的 MVP 陷阱

| 陷阱 | 表现 | 解法 |
|------|------|------|
| 完美主义 | "这个界面不够好看" | V1.0 只要能用，V1.1 优化体验 |
| 防御性设计 | "万一用户量大了扛不住" | 先获取用户，再优化性能 |
| 竞品对标 | "竞品有这个功能我们也得有" | 先做差异化功能，通用功能后面再加 |
| 边缘场景 | "如果用户输入了特殊字符怎么办" | 先处理主流场景，边缘情况用错误提示兜底 |
| 管理功能 | "需要后台管理系统" | 先用数据库直接操作或简单脚本代替 |

### MVP 自检问题

在确定 MVP 范围后，问自己：

1. 如果只能留 3 个功能，你会留哪 3 个？→ 这就是真正的 P0
2. 用户拿到这个产品后，第一分钟会做什么？→ 这个流程必须顺畅
3. 如果必须在 2 周内上线，你会砍掉什么？→ 被砍掉的就不是 MVP

## 复杂度评估框架

### 复杂度分级

| 级别 | 时间范围 | 典型特征 | 示例 |
|------|---------|---------|------|
| **简单** | < 3 天 | 纯前端变更、配置调整、已有组件组合 | 添加一个筛选条件、修改文案、调整布局 |
| **中等** | 3-7 天 | 新增 API 端点、中等业务逻辑、需要测试 | 新增一个列表页、实现基本的 CRUD、对接一个简单 API |
| **复杂** | 1-2 周 | 新架构模块、第三方集成、性能优化 | 实现支付流程、对接复杂第三方 API、搜索功能 |
| **极复杂** | > 2 周 | 底层架构变更、高性能要求、分布式系统 | 实时协作编辑、大规模数据处理、多租户架构 |

### 复杂度评估维度

对每个功能从 5 个维度评估：

| 维度 | 低 (1分) | 中 (2分) | 高 (3分) |
|------|---------|---------|---------|
| **逻辑复杂度** | 简单 CRUD | 有条件分支和状态管理 | 复杂工作流、并发处理 |
| **依赖项** | 无外部依赖 | 依赖 1-2 个内部模块 | 依赖外部 API 或新基础设施 |
| **不确定性** | 做过类似的 | 有参考但需要适配 | 没做过，需要研究 |
| **测试难度** | 单元测试即可 | 需要集成测试 | 需要端到端测试 + 性能测试 |
| **影响范围** | 独立模块 | 影响 2-3 个模块 | 影响整个系统架构 |

总分 5-7 → 简单，8-10 → 中等，11-13 → 复杂，14-15 → 极复杂。

### 极复杂功能的处理

当一个功能被评为"极复杂"时：

1. **必须做架构评审**：拉 CTO 参与，确认技术方案
2. **尝试拆分**：能否拆成 2-3 个较小的功能分批做？
3. **评估 ROI**：这个功能的价值是否配得上 2 周+ 的投入？
4. **考虑延后**：能否先用简化方案，后续再做完整版？

## 技术债务管理

### 什么时候可以走捷径

| 情况 | 决策 | 理由 |
|------|------|------|
| MVP 阶段验证假设 | ✅ 可以走捷径 | 假设可能被推翻，精心设计可能浪费 |
| 确定要长期维护的核心模块 | ❌ 不要走捷径 | 技术债务会越积越贵 |
| 竞争窗口紧迫 | ✅ 可以走捷径 | 先上线再优化，但要记录债务 |
| 涉及安全和数据的模块 | ❌ 不要走捷径 | 安全债务的修复成本极高 |
| 一次性的运营活动 | ✅ 可以走捷径 | 用完即弃，不值得过度设计 |

### 技术债务记录格式

```markdown
| 债务项 | 当前状态 | 影响 | 修复成本 | 建议处理时间 |
|--------|---------|------|---------|-------------|
| API 没有限流 | 裸奔 | 高并发时服务崩溃 | 中等 (3天) | V1.1 |
| 数据库没有索引优化 | 慢查询 | 数据量 > 1万时变慢 | 简单 (1天) | V1.1 |
| 前端没有错误边界 | 报错白屏 | 用户体验差 | 简单 (2天) | V1.1 |
| 没有自动化测试 | 手动测试 | 回归成本高 | 复杂 (1周) | V1.2 |
```

### 债务清理节奏

- **V1.0 后**：记录所有已知债务，评估影响
- **V1.1**：处理影响用户体验的高优先级债务（20% 时间）
- **V1.2**：处理影响开发效率的债务 + 架构层面优化（20% 时间）
- **每个周期**：至少预留 20% 的开发时间用于债务清理

## 版本发布标准

### 发布前检查清单

#### V1.0 发布标准（MVP）

- [ ] 所有 P0 功能已完成并通过验收标准
- [ ] 核心流程无阻塞性 Bug
- [ ] 基本的错误处理已覆盖（不会白屏/崩溃）
- [ ] 有最基础的用户引导（用户知道怎么开始）
- [ ] 数据采集已部署（能跟踪核心指标）
- [ ] 有反馈收集渠道（用户能报告问题）

#### V1.1 发布标准

- [ ] V1.0 中发现的 P0 级 Bug 全部修复
- [ ] 用户反馈最集中的 Top 3 体验问题已优化
- [ ] 选定的 P1 功能已完成并通过验收
- [ ] 核心功能有自动化测试覆盖
- [ ] A/B 测试框架就绪（至少对 1 个功能）

#### V1.2 发布标准

- [ ] 增长相关功能已完成（分享、邀请、SEO 等）
- [ ] 主要技术债务已处理
- [ ] 性能指标达到目标（加载时间、响应速度）
- [ ] 监控和报警系统完善
- [ ] 下一个周期的规划文档已就绪

### 发布流程

```
代码冻结 → 回归测试 → 灰度发布（10%用户）→ 监控 24h → 全量发布
```

## 迭代反馈循环

### 数据驱动迭代模型

```
V1.0 发布
    ↓
收集数据（1-2 周）
    ↓
分析数据 → 四个象限分类
    ↓
V1.1 规划
    ↓
V1.1 开发 + 发布
    ↓
收集数据（1-2 周）
    ↓
V1.2 规划 ...
```

### 数据分类四象限

| | 数据好 | 数据差 |
|------|--------|--------|
| **用户反馈好** | 继续保持，作为增长杠杆 | 检查数据采集是否准确 |
| **用户反馈差** | 功能有价值但体验需优化 | 重新审视是否该保留 |

### V1.0 → V1.1 的关键数据

| 指标 | 数据来源 | 决策影响 |
|------|---------|---------|
| 激活率 | 注册 → 完成核心操作的比例 | < 30% → 需要优化 onboarding |
| 核心功能使用率 | 各 P0 功能的使用频率 | 最低的功能 → 考虑是否真的 P0 |
| 完成率 | 开始核心流程 → 完成的比例 | < 50% → 流程有阻塞点需要定位 |
| 留存率 | Day 1 / Day 7 回访 | Day 1 < 40% → 核心价值传递有问题 |
| 用户反馈 | NPS、评论、客服工单 | 高频问题 → V1.1 优先修复 |

### V1.1 → V1.2 的关键数据

| 指标 | 数据来源 | 决策影响 |
|------|---------|---------|
| 留存曲线 | Day 7 / Day 14 / Day 30 | 曲线是否趋于平稳（产品有粘性） |
| 功能采用率 | V1.1 新功能的使用情况 | 低采用率 → 功能发现性或价值有问题 |
| 增长指标 | 自然增长率、推荐率 | 推荐率 > 20% → 有口碑传播基础 |
| 付费转化 | 免费 → 付费的转化率 | < 5% → 定价或价值感知需要调整 |
| 性能数据 | 加载时间、错误率 | 劣化趋势 → V1.2 必须处理技术债务 |

### 反馈收集方法

| 方法 | 时机 | 适用于 |
|------|------|--------|
| 应用内反馈按钮 | 随时 | 收集使用中的即时反馈 |
| NPS 调查 | V1.0 发布后 2 周 | 衡量整体满意度 |
| 用户访谈 | V1.0 发布后 1-2 周 | 深入理解使用行为和痛点 |
| 数据分析 | 持续 | 发现用户说不出口的问题 |
| 客服工单分析 | 持续 | 发现高频阻塞问题 |
| 竞品用户社区 | 持续 | 发现行业趋势和未满足需求 |

### 迭代决策模板

每个版本规划前，填写此表：

```markdown
## V1.X 迭代决策

### 上个版本的核心发现
1. [数据发现1]：[影响]
2. [数据发现2]：[影响]
3. [用户反馈1]：[影响]

### 本版本要回答的问题
1. [假设/问题]
2. [假设/问题]

### 本版本范围决策
| 功能 | 来源 | 理由 | 预期指标影响 |
|------|------|------|-------------|
| 功能A | V1.0 数据 → 激活率低 | 优化 onboarding 流程 | 激活率 +15% |
| 功能B | 用户反馈 Top 1 | 解决最频繁的抱怨 | NPS +10 |
| 功能C | PRD P1 | 增强核心体验 | 留存 +5% |
```
