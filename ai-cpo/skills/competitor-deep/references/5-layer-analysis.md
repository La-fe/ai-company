# 5 层竞品深度分析方法论

> 本文档为 cpo-competitor-deep 的核心参考，在 Phase 1 开始时加载。
> 定义每层的分析方法、问题清单、深度标准和常见陷阱。

---

## 框架总览

5 层框架把一个竞品的理解从"知道名字"推进到"完全理解其产品架构"。每层的产出是下层的输入：

```
Layer 1: 问题空间  →  "谁在什么场景下用这个产品"
    ↓ 输出: 用户画像 + JTBD → 决定 Layer 2 的实体范围
Layer 2: 数据模型  →  "产品用哪些实体来解决问题"
    ↓ 输出: 实体关系图 → 决定 Layer 3 的旅程步骤
Layer 3: 用户旅程  →  "用户怎么跟这些实体交互"
    ↓ 输出: 旅程步骤 → 暴露 Layer 4 的技术需求
Layer 4: 技术能力  →  "用了什么技术/AI 来支撑"
    ↓ 输出: 能力清单 → 影响 Layer 5 的定价和 Feature
Layer 5: Feature+商业 → "最终卖什么、怎么赚钱"
```

**核心原则**: 上层约束下层。如果 Layer 1 的用户画像是"个人创作者"，Layer 5 的定价就不应该出现"企业套餐"——除非有明确数据支持。

---

## Layer 1: 问题空间 — 详细方法

### 目标

理解竞品为谁解决什么问题，以及用户在什么场景下被触发使用。

### 问题清单

**用户画像（必答）：**
- [ ] 目标用户是谁？（具体特征，不是泛称）
- [ ] 用户的规模/阶段？（粉丝数、收入水平、团队大小）
- [ ] 用户的技术能力？（会不会写代码、用过什么工具）
- [ ] 竞品官网 hero copy 如何描述目标用户？

**触发场景（必答）：**
- [ ] 什么事件让用户第一次搜索这类产品？
- [ ] 用户在使用前处于什么状态？（痛苦/好奇/被推荐）
- [ ] 竞品的获客渠道暗示了什么触发场景？

**现有替代方案（必答）：**
- [ ] 用户之前怎么做同样的事？
- [ ] 替代方案的痛点是什么？（竞品官网 vs alternatives 部分可提取）

**JTBD 三维度（必答）：**
- [ ] 功能性任务：用户要完成什么操作？
- [ ] 情感性任务：用户想要什么感受？（自信/省心/专业）
- [ ] 社会性任务：用户想要什么形象？（被粉丝认可/看起来专业）

### 深度标准

**足够**：用户画像具体到可以想象出一个真实的人，JTBD 三维度每个都有具体描述，触发场景可以讲一个故事。

**不足**：用户画像是泛称（"创作者"、"小企业主"），JTBD 只有功能性维度，触发场景只是"想要这个产品"。

### 信息来源

| 来源 | 提取内容 | 可靠度 |
|------|---------|--------|
| 竞品官网 hero copy | 目标用户描述、核心价值主张 | 高（官方定位） |
| 竞品 About/Mission 页 | 创始故事、目标用户群 | 高 |
| 用户评价（G2/Capterra/ProductHunt） | 真实用户画像、使用场景 | 高（用户视角） |
| 竞品社媒案例/博客 | 成功案例的用户特征 | 中 |
| 竞品广告/推广内容 | 获客渠道暗示触发场景 | 中 |

## Layer 2: 数据模型 — 详细方法

### 目标

提取竞品的核心实体和关系，理解其产品架构的"骨架"。

### 问题清单

**实体清单（必答）：**
- [ ] 竞品有哪些核心实体？（从 API/UI/定价页提取，详见 entity-extraction.md）
- [ ] 每个实体的类型？（核心/辅助/价值/事务）
- [ ] 有没有 AI 原生实体？（只因 AI 存在而存在的实体）

**实体关系（必答）：**
- [ ] 实体之间的关系类型？（1:N, N:N, 包含, 依赖）
- [ ] 数据流方向？（A 创建后触发 B 更新）
- [ ] 是否画了关系图（ASCII），不只是列表？

**定价边界（必答）：**
- [ ] 哪些实体/功能在 Free 层可用？
- [ ] 哪些是付费解锁的？
- [ ] 付费的边界是数量限制还是功能限制？

### 深度标准

**足够**：有实体清单 + 关系图 + 定价映射，关系图标注了关系类型，能看出数据如何流转。

**不足**：只有实体名字列表，没有关系图，或者关系图只是层级树而不是交互图。

### 常见陷阱

| 陷阱 | 表现 | 纠正 |
|------|------|------|
| 只列实体名不画关系 | 实体清单但无关系图 | 必须画 ASCII 关系图，标注关系类型 |
| 层级树代替关系图 | User → Store → Product 的树形结构 | 标注关系方向和类型，如 User -1:N-> Store -1:N-> Product |
| 忽略定价边界 | 只列实体不区分免费/付费 | 每个实体标注在哪个定价层可用 |
| AI 实体遗漏 | 有 AI 功能但没提取相关实体 | 检查是否有 Credit/Generation/Prompt 等实体 |

## Layer 3: 用户旅程 — 详细方法

### 目标

还原用户从发现产品到持续使用的完整体验，每步标注涉及的实体和操作。

### 问题清单

**完整旅程（必答）：**
- [ ] 发现阶段：用户怎么知道这个产品？
- [ ] 注册阶段：注册流程有几步？需要什么信息？
- [ ] Onboarding 阶段：首次使用引导是什么？
- [ ] 核心使用阶段：用户日常怎么使用？
- [ ] 留存阶段：什么让用户持续回来？
- [ ] 每步标注了涉及的实体和 CRUD 操作？

**Onboarding 详解（必答）：**
- [ ] 首次登录后的每一步是什么？
- [ ] 有没有模板/引导/问答？
- [ ] 多快能感受到价值？（TTFV 估算）

**最小价值旅程（必答）：**
- [ ] 从注册到感受价值的最短路径有几步？
- [ ] 哪些步骤可以跳过？

### 深度标准

**足够**：完整旅程表覆盖发现→留存全阶段，每步有实体/CRUD 标注，Onboarding 有逐步详解，有 TTFV 估算。

**不足**：旅程只到注册或 Onboarding，没有核心使用和留存阶段；没有实体/CRUD 标注。

### 常见陷阱

| 陷阱 | 表现 | 纠正 |
|------|------|------|
| 旅程只到 Onboarding | "注册→设置→完成" | 必须覆盖到留存阶段，包括复访触发 |
| 缺少实体标注 | "用户创建商品" 但没标注 Product 实体 | 每步必须标注涉及哪些 Layer 2 实体 |
| 忽略分支路径 | 只有主流程 | 标注关键分支（如跳过 Onboarding 的路径） |
| TTFV 缺失 | 没有估算首次价值感知时间 | 必须给出 TTFV 估算（分钟/步骤数） |

## Layer 4: AI/技术能力 — 详细方法

### 目标

理解竞品使用了什么技术/AI 来支撑产品体验，评估可靠度和人机协作设计。

### 问题清单

**AI 功能分析（如果竞品有 AI）：**
- [ ] AI 做了什么？（生成/推荐/分析/自动化）
- [ ] 可靠度估计？（可靠 >95% / 中等 70-95% / 实验 <70%）
- [ ] HITL 设计？（全自动/人审核/人协作/纯人工）
- [ ] AI 失败时用户的退路是什么？

**技术架构（通用）：**
- [ ] 有没有公开 API？API 设计风格？
- [ ] 集成了哪些第三方服务？（支付/社媒/邮件/分析）
- [ ] 有没有技术栈线索？（前端框架、API 风格）

**竞品无 AI 时：**
- [ ] 哪些环节未来可能引入 AI？
- [ ] 当前用什么方式做 AI 可以替代的工作？

### 深度标准

**足够**：AI 功能有清单 + 可靠度评估 + HITL 标注；技术集成有具体服务名；有 AI 失败退路分析。

**不足**：只说"用了 AI"但没分析可靠度和 HITL；技术集成只说"有支付"但不说具体是 Stripe 还是 PayPal。

## Layer 5: Feature + 商业模型 — 详细方法

### 目标

列出竞品的完整 Feature 清单和商业模式，理解其盈利逻辑和留存策略。

### 问题清单

**Feature 清单（必答）：**
- [ ] 完整功能列表，按类别组织
- [ ] 每个功能标注核心/增值/差异化
- [ ] 哪些功能是竞品独有的？

**定价模型（必答）：**
- [ ] 有几个定价层？
- [ ] 每层的价格、目标用户、包含的 Feature？
- [ ] 免费层有什么限制？

**留存机制（必答）：**
- [ ] 数据锁定：用户的数据/内容能不能迁出？
- [ ] 收入依赖：用户是否通过平台赚钱？（停用 = 失去收入）
- [ ] 网络效应：用户越多价值越高吗？
- [ ] 内容积累：用户投入的内容/时间会让离开成本增加吗？
- [ ] 每个机制标注了哪些实体支撑 + 强度评估

**变现逻辑（必答）：**
- [ ] 商业模式类型？（订阅/交易费/Credit/混合）
- [ ] 经济可行性？（用户 LTV vs CAC 的暗示）

### 深度标准

**足够**：Feature 清单有分类和标注，定价有完整对比表，留存有具体机制和实体映射，变现有逻辑总结。

**不足**：Feature 只是功能名列表；定价只有价格数字没有 Feature 映射；留存只说"用户会留下来"。

## 跨层一致性检查

完成 5 层分析后，进行以下检查：

| 检查项 | 方法 |
|--------|------|
| Layer 1 用户画像 vs Layer 5 定价 | 画像中的用户能承受这个价格吗？ |
| Layer 2 实体 vs Layer 3 旅程 | 旅程中的每个实体都在 Layer 2 定义了吗？ |
| Layer 3 旅程 vs Layer 4 技术 | 旅程中涉及 AI 的步骤在 Layer 4 都分析了吗？ |
| Layer 2 定价边界 vs Layer 5 定价 | 定价映射一致吗？ |
| Layer 4 AI 能力 vs Layer 5 Feature | AI 功能在 Feature 清单中都覆盖了吗？ |
