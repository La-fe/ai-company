# 周期规划方法论

## 1. 周期定义

**默认周期**: 1 个月 = 4 周

```
周期 = 一个可独立评估成败的最小战略单元
```

### 为什么是 1 个月

| 周期长度 | 优点 | 缺点 | 适用 |
|---------|------|------|------|
| 1 周 | 反馈快 | 战略性任务做不完 | 执行层 |
| **1 个月** | **平衡深度和速度** | **需要周维度拆解** | **战略+执行** |
| 1 季度 | 战略充分 | 反馈太慢，纠偏成本高 | 大公司 |

### 周期结构

```
Month Cycle (4 weeks)
├── Week 1: 详细执行计划（每天级别）
├── Week 2: 关键指标 + 里程碑
├── Week 3: 关键指标 + 里程碑
└── Week 4: 关键指标 + 复盘准备
```

---

## 2. 第一周详细规划标准

第一周是唯一需要"日级别"规划的周。原则：**只规划看得清的部分**。

### 日计划模板

```markdown
### Day {N} — {日期}

**核心目标**: {当天最重要的 1 件事}

| 任务 | 维度 | 预估时长 | 产出物 | 完成标准 |
|------|------|---------|--------|---------|
| xxx  | 产品 | 2h      | xxx    | xxx     |

**阻塞风险**: {如果有的话}
```

### 规划规则

1. 每天 P0 任务 ≤ 2 个
2. 每天实际可执行时间按 6 小时算（留 buffer）
3. 高认知负荷任务放上午
4. 会议和沟通集中在下午
5. 有依赖的任务不排在同一天

---

## 3. 第 2-4 周指标规划标准

只需定义：

| 要素 | 说明 | 示例 |
|------|------|------|
| 周目标 | 本周结束时要达到的状态 | "V1.0 核心功能开发完成" |
| 关键指标 | 2-3 个可量化指标 | DAU 100, 注册转化 5% |
| 里程碑 | 本周必须交付的产出物 | PRD 评审通过、首批广告上线 |
| 风险预判 | 可能阻塞的事项 | 竞品发布类似功能 |

### 周指标模板

```markdown
### Week {N}

**目标**: {一句话概括}
**指标**:
- [ ] {指标 1}: 当前值 → 目标值
- [ ] {指标 2}: 当前值 → 目标值

**里程碑**:
- [ ] {里程碑 1} — 负责人
- [ ] {里程碑 2} — 负责人

**风险**: {预判}
```

## 4. 输出目录结构

```
{output_base}/{company}/
│
├── context/                          # 共性层：公司 DNA（跨规划持久化）
│   ├── company.md                    # 身份/产品/市场/定位
│   ├── arsenal.md                    # 武器库
│   └── competitors/                  # 竞品档案（持续积累）
│       └── {name}.md
│
├── plans/                            # 规划层：所有规划单元
│   ├── 2026-03/                      # 月度周期 (type: cycle)
│   │   ├── plan.md                   # 统一入口
│   │   ├── goals.md                  # 本周期目标
│   │   ├── tasks.md                  # CEO 任务拆解
│   │   ├── prd.md                    # CPO PRD
│   │   ├── version-plan.md
│   │   ├── growth-plan.md
│   │   ├── roadmap.md
│   │   └── weekly/
│   │       ├── week-1.md             # 日级别
│   │       └── week-2~4.md           # 指标+里程碑
│   │
│   ├── 2026-03-15-{name}/            # 临时计划 (type: initiative)
│   │   ├── plan.md
│   │   └── {按需产出}.md
│   │
│   └── 2026-03-20-{name}/            # CXO 请求 (type: request)
│       ├── plan.md
│       └── {CXO产出}.md
│
└── todos.md                          # 日常层：滚动待办（跨规划）
```

### 三种规划类型

| 类型 | 命名规则 | 包含文件 | 生命周期 |
|------|---------|---------|---------|
| cycle | `YYYY-MM` | 全套（goals→tasks→prd→growth→roadmap→weekly） | 固定 4 周 |
| initiative | `YYYY-MM-DD-{name}` | 按需（只包含需要的 CXO 产出） | 几天到几周 |
| request | `YYYY-MM-DD-{name}` | 最少（plan.md + 单个 CXO 产出） | 一次性 |

### 文件生命周期

| 文件 | 层级 | 创建时机 | 更新频率 |
|------|------|---------|---------|
| company.md | 共性 | /init-company | 季度 |
| arsenal.md | 共性 | /init-company | 持续（新 skill 时） |
| competitors/*.md | 共性 | /cpo-plan | 持续积累 |
| plan.md | 规划 | 规划创建时 | 每次变更追加记录 |
| goals.md | 规划 | /ceo-plan | 周期内更新 |
| tasks.md | 规划 | /ceo-plan | 每周 review |
| weekly/*.md | 规划 | /ceo-plan | Week 1 详细，后续按需 |
| todos.md | 日常 | 首次使用 | 随时 |

---

## 5. 周期启动 Checklist

新周期开始前必须确认：

- [ ] 上个周期复盘已完成
- [ ] goals.md 已更新为新周期目标
- [ ] arsenal.md 中的 skill 状态是最新的
- [ ] 团队资源可用性已确认
- [ ] 外部依赖（服务商、合作方）已对齐

---

## 6. 周期切换规则

```
当前周期结束 →
  1. CEO review（/ceo-plan --mode review）
  2. 未完成 P0 自动滚入下周期
  3. 未完成 P1/P2 重新评估（保留 or 放弃）
  4. 创建新周期目录（plans/YYYY-MM/）
  5. 归档相关 initiative（更新 plan.md 状态为 completed/cancelled）
```

---

## 7. plan.md 模板

每个规划单元的统一入口文件：

```markdown
# Plan — {名称}

> 类型: cycle | initiative | request
> 创建: YYYY-MM-DD
> 状态: planning | active | completed | cancelled
> 紧急度: P0 | P1 | P2

## 目标
{一句话：要达成什么}

## 触发
{什么事件/输入触发了这个计划}

## CXO 处理链
- [ ] CEO: 目标拆解 → tasks.md
- [ ] CPO: PRD 生成 → prd.md
- [ ] CMO: 增长策略 → growth-plan.md

## 变更记录
| 日期 | 变更 | 原因 |
|------|------|------|
```
